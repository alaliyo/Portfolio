# 랑구스토리

한국어 교원들을 위한 AI 교안 생성 플랫폼 프론트엔드 프로젝트
사용자 페이지와 관리자 페이지를 함께 개발하며, **Feature-Sliced Design(FSD)을 베이스로** 유지보수성과 확장성을 고려한 구조를 정리·적용한 프로젝트입니다.

---

## 프로젝트 소개

랑구스토리는 교육 도메인의 웹 서비스로, 저는 이 프로젝트에서 **사용자 페이지와 관리자 페이지 프론트엔드 개발**에 참여했습니다.
기능이 점진적으로 확장되는 프로젝트 특성을 고려해, 단기 구현보다 **지속적으로 유지보수 가능한 구조**를 만드는 데 집중했습니다.

코드와 세부 기능은 공개할 수 없어, 이 문서에서는 **FSD 기반의 아키텍처 적용 방향, 폴더 구조 원칙, 상태 관리 기준, 협업을 위한 컨벤션** 중심으로 정리했습니다.

- **기간**: 2025.11 ~ 2026.03
- **역할**: 프론트엔드 개발
- **범위**:
  - 사용자 페이지 개발
  - 관리자 페이지 개발
  - 사용자 페이지 설계 (기능 정의서, 아키텍처, 코드 컨벤션 등)
  - 관리자 페이지 설계 (기능 정의서, 아키텍처, 코드 컨벤션 등)
  - 상태 관리 및 데이터 흐름 정리
  - 공통 UI 및 개발 규칙 정리

---

## 사용 기술

- **Core**: React, TypeScript, Vite
- **Routing**: react-router-dom
- **Client State**: Zustand
- **Server State**: TanStack Query
- **API**: Axios
- **Validation**: zod
- **Styling**: styled-components

## 설계 배경

이 프로젝트는 사용자 영역과 관리자 영역이 함께 존재하고, 기능이 계속 추가되는 구조였습니다.
이런 형태의 서비스는 초기 구현 속도만 우선하면 다음과 같은 문제가 생기기 쉽다고 판단했습니다.

- 페이지 단위로 코드가 커지면서 책임이 섞이는 문제
- API 호출, 상태 처리, UI 로직이 한곳에 집중되는 문제
- 공통 요소와 도메인별 기능의 경계가 흐려지는 문제
- 기능이 늘어날수록 영향 범위를 예측하기 어려운 문제

이런 문제를 줄이기 위해, **FSD의 레이어 분리 방식과 기능 중심 사고를 참고해** 화면 단위보다 **기능 단위로 책임을 분리하는 구조**를 적용했습니다.

---

## 아키텍처 방향

### 1. 레이어 분리

프로젝트 구조는 **Feature-Sliced Design(FSD)을 기반으로**, 역할에 따라 레이어를 나누고 의존성 방향이 뒤섞이지 않도록 정리했습니다.

```
app → pages → widgets → features → entities → shared
```

각 레이어는 다음 역할을 가집니다.

- **app**: 앱 초기화, 전역 설정, 라우팅, Provider 구성
- **pages**: 라우트와 직접 연결되는 화면 단위
- **widgets**: 페이지를 구성하는 큰 UI 블록
- **features**: 사용자 액션 단위 기능
- **entities**: 도메인 모델과 데이터 구조
- **shared**: 공통 UI, 유틸, 설정, API 클라이언트

이 구조를 통해 화면 구성, 비즈니스 로직, 공통 자산의 책임을 분리하고자 했습니다.

---

### 2. 기능 중심 구조

페이지 기준으로 로직을 모으기보다, **FSD의 기능 중심 구조를 바탕으로 사용자 행동 단위**로 기능을 분리했습니다.

예를 들어 폼 처리, 인증 흐름, 데이터 조회/수정 같은 기능은 하나의 페이지 내부에 흩어두지 않고, 기능 단위로 모아 관리할 수 있게 구성했습니다.

이 방식으로 기대한 효과는 다음과 같습니다.

- 기능별 책임 범위가 명확해짐
- 수정 시 영향 범위 파악이 쉬워짐
- 유사한 패턴을 재사용하기 쉬워짐
- 협업 시 구조 이해 비용이 줄어듦

---

### 3. 상태 관리 기준 분리

상태도 목적에 따라 나눠 관리했습니다.

- **TanStack Query**: 서버 데이터 조회, 캐싱, 비동기 요청 상태
- **Zustand**: UI 흐름, 로컬 상호작용, 전역 클라이언트 상태

이렇게 분리해 서버 상태와 화면 상태가 서로 섞이지 않도록 했고,
데이터 흐름을 더 예측 가능하게 유지하려고 했습니다.

---

### 4. API 계층 분리

화면 컴포넌트가 직접 API 처리 로직을 가지지 않도록,
통신 계층과 조회/수정 계층의 역할을 나누는 방향으로 구조를 정리했습니다.

또한 입력값과 데이터 형태를 보다 명확하게 다루기 위해 검증 로직을 분리해 관리할 수 있도록 구성했습니다.

## 폴더 구조 원칙

```
src/
  app/
  pages/
  widgets/
  features/
  entities/
  shared/
  main.tsx
```

### app

앱 초기화, 라우팅, 전역 Provider, 공통 레이아웃 등 전역 설정을 담당합니다.

### pages

라우터와 직접 연결되는 페이지 단위입니다.
페이지는 실제 비즈니스 로직을 과도하게 담기보다, 필요한 기능과 UI를 조합하는 역할에 집중합니다.

### widgets

페이지를 이루는 큰 UI 블록을 담당합니다.
페이지에서 반복되거나 독립적으로 조립 가능한 UI 단위를 관리합니다.

### features

사용자 액션 단위 기능을 담당합니다.
기능별 UI, 상태, 데이터 처리 로직을 함께 묶어 응집도를 높이는 방향으로 관리했습니다.

### entities

도메인 모델과 데이터 구조를 담당합니다.
특정 화면보다는 서비스 전반에서 공통으로 다뤄지는 개념을 정리하는 계층입니다.

### shared

공통 UI 컴포넌트, 유틸 함수, 설정, 스타일, API 클라이언트 등
여러 레이어에서 함께 사용하는 자산을 관리합니다.

---

## 네이밍 및 관리 규칙

프로젝트 전반에서 파일과 폴더의 역할을 빠르게 파악할 수 있도록 기본 규칙을 정리해 사용했습니다.

### 폴더 규칙

- 도메인/기능 폴더: `camelCase`
- UI 컴포넌트 폴더: `PascalCase`

### 파일 규칙

- 컴포넌트: `PascalCase.tsx`
- 훅: `useName.ts`
- API: `name.api.ts`
- 스키마: `name.schema.ts`
- 스토어: `useName.store.ts`
- 쿼리: `useName.query.ts`
- 뮤테이션: `useName.mutation.ts`
- 타입: `name.type.ts`
- 상수: `name.constant.ts`
- 스타일: `name.style.ts`
- 공개 export: `index.ts`

이 규칙은 단순한 형식 통일보다는,
파일 이름만 보고도 역할을 빠르게 이해할 수 있도록 하는 데 목적이 있습니다.

## 회고

이 프로젝트는 단순히 화면을 구현하는 데서 끝나는 것이 아니라,
기능이 계속 늘어나는 서비스에서 **어떤 구조가 장기적으로 유지보수에 유리한가**를 고민하게 만든 경험이었습니다.

특히 프론트엔드에서도 구조 설계와 책임 분리가 개발 생산성과 협업 효율에 큰 영향을 준다는 점을 다시 확인할 수 있었고,
앞으로도 기능 구현과 함께 **기존 아키텍처 원칙을 잘 이해하고, 프로젝트에 맞게 적용하는 개발 방식**을 더 중요하게 가져가고 싶습니다.
